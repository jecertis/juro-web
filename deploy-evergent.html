Installation Guide

April 2026  ·  Shared privately — not for redistribution
Non-custodial by design. The Juro agent runs entirely inside your AWS account under a read-only IAM role you provision and control. No scan data, no infrastructure state, and no findings ever leave your perimeter. Juro receives only what you choose to publish — a hash and signature — to an append-only transparency log.

Before you begin

  • Docker and Docker Compose v2+ installed on the host that will run the agent stack.
  • AWS CLI v2 configured with credentials that have permission to create an IAM role and attach an inline policy.
  • Node.js 18+ to install and run the juro-deploy CLI.
  • Signed DPA on file with Juro. The agent must not be run before the data processing agreement is signed. Contact hello@jurocompliant.com if you need it resent.
  • Rule-pack SHA and image digest provided by Juro (sent alongside this guide). You will paste these into juro-deploy.yaml in Step 1.

Run the host prerequisite check at any time:

juro-deploy doctor
Step 0

Provision the IAM role

Create a role named juro-agent in your AWS account and attach the inline policy below. The trust policy should allow EC2, ECS, or whichever compute type you will use to assume the role. The agent authenticates via the role attached to the instance — no static credentials, ever.

Step 1

Create juro-deploy.yaml

Create this file in your infrastructure repo. Replace the placeholder values with what Juro sent you and your own ARNs. Commit it to source control — it is the versioned record of your engagement configuration.

juro-deploy.yaml
engagement_slug: your-org-aws           # lowercase kebab-case, e.g. evergent-aws
cloud: aws
rule_pack_sha: "sha256:JURO_PROVIDED"   # from the digest sheet Juro sent you
scanner_image_digest: "sha256:JURO_PROVIDED"
signer_key_id: "arn:aws:kms:REGION:ACCOUNT:key/YOUR_KEY_ID"
tl_endpoint: "https://transparency-log.juro.dev"
agent_role_arn: "arn:aws:iam::ACCOUNT:role/juro-agent"

scope:
  services:
    - rds
    - s3
    - lambda
    - dynamodb
    - secretsmanager
    - ssm
    - cloudfront
    - apigateway
    - vpc
  regions:
    - us-east-1         # add every region where you run regulated workloads
    - ap-south-1        # ap-south-1 and ap-south-2 map to DPDP automatically

# Optional — bring your own LLM for triage (recommended if you have a GPU instance).
# Without this, the stack starts a local Ollama sidecar inside the compose network.
# triage_llm_url: "http://your-gpu-host:11434"
# triage_model: "llama3.2:3b"
Step 2

Initialise the stack

Pulls the pinned agent image, renders a Docker Compose file, and starts the stack. Creates a working directory at ~/juro-engagement/<slug>/.

juro-deploy init --config juro-deploy.yaml
✓ image pulled: ghcr.io/jecertis/cloud-scanner@sha256:…
✓ compose written: ~/juro-engagement/your-org-aws/docker-compose.yml
✓ stack started — scanner listening on localhost:3100
Step 3

Run preflight checks

Validates that all 40 required IAM actions are allowed under agent_role_arn and that the transparency log endpoint is reachable. Do not proceed until every check passes.

juro-deploy preflight --slug your-org-aws
✓ iam:all-actions — 40 actions simulated, all ALLOWED
✓ egress:transparency-log — HTTP 200
— oidc:token-exchange — deferred (M5)
— artifact-store:write — deferred (M5)
preflight: 2 passed, 2 skipped

If any IAM check fails, open the AWS IAM console, locate the juro-agent role, and confirm the inline policy matches the JSON in Step 0 exactly. Common cause: a service-control policy (SCP) at the organisation level overriding the allow.

Step 4

Run the discovery scan

Sweeps all configured services and regions using read-only API calls. Writes a residency map — a structured inventory of every resource the agent found — to your working directory. No rule pack is active at this stage; this is a pure discovery pass.

juro-deploy scan --slug your-org-aws
✓ scan complete — 247 resources discovered
residency_map.json → ~/juro-engagement/your-org-aws/residency_map.json
Review required before proceeding. Open residency_map.json and confirm the listed services and regions match what you expect. If regulated workloads are missing, widen the scope in juro-deploy.yaml and re-run. Your compliance team should sign off on this file before you move to rule assignment.
Step 5

Generate the rule pack suggestion

Reads the residency map and maps each region to its applicable regulation: ap-south-1 and ap-south-2 → DPDP; eu-* regions → GDPR. DORA is not auto-detected — add it manually if it applies to your organisation. Writes a draft rules.yaml for your review.

juro-deploy rules suggest --slug your-org-aws
rules.yaml → ~/juro-engagement/your-org-aws/rules.yaml
Edit before signing. Open rules.yaml and replace every sha256:PENDING-* placeholder with the real rule-pack digest Juro sent you. Add DORA if required. For any resource listed under unmapped_surfaces, assign it to a regulation manually. Commit the edited file to source control before running the next step.
Step 6

Sign the configuration

Computes a deterministic hash of rules.yaml and writes a signing bundle. This bundle is the compliance evidence artifact — the tamper-evident record that a specific, reviewed rule configuration was committed at a specific point in time.

juro-deploy config sign rules.yaml --slug your-org-aws --output artifacts/signing-bundle.json
config_hash: sha256:7e1b2c3d…
✓ signing-bundle.json written to artifacts/

The current signing status is hash_committed. KMS-backed cryptographic signing (using the signer_key_id KMS key you configured) ships in the next milestone. Store signing-bundle.json in your artifact store — it is the record you produce for an auditor.

Frequently asked questions

Does Juro see our cloud data?

No. The agent runs inside your AWS account and makes read-only API calls directly to AWS. It never proxies traffic through Juro infrastructure. Juro has no network path into your VPC and no credentials that would allow one.

The only thing that leaves your environment is a hash and a digital signature — not findings, not resource names, not configuration values. Even those are optional: you control whether the signing bundle is published to the transparency log.

What data leaves our environment?

By default: nothing. All scan artifacts — the residency map, rules.yaml, and the signing bundle — are written to ~/juro-engagement/<slug>/ on the host that runs the agent.

If you configure a transparency log endpoint (tl_endpoint), only the hash and signature of the signing bundle are published — never the content of the bundle itself. The transparency log entry proves the artifact existed at a point in time without revealing what it contains.

There is no telemetry channel, no call-home mechanism, and no background process that contacts Juro servers.

My preflight failed on an IAM check — what do I do?

The most common cause is a Service Control Policy (SCP) at the AWS organisation level that denies the action before the role policy can allow it. Check your organisation's SCPs in AWS Organizations and confirm none of them deny the failing action.

If there is no SCP issue, open the juro-agent role in IAM and confirm the inline policy matches the JSON in Step 0 exactly. A missing action or a resource-level restriction (e.g., "Resource": "arn:aws:s3:::specific-bucket" instead of "*") will cause a simulation failure.

Re-run juro-deploy preflight --slug your-org-aws after any policy change.

Which AWS regions are supported?

Any AWS region can be added to the scope.regions list in juro-deploy.yaml. There is no agent-side restriction on regions.

For regulation auto-detection: ap-south-1 and ap-south-2 map to DPDP; all eu-* regions map to GDPR. Resources in other regions will appear under unmapped_surfaces in rules.yaml — assign them to a regulation manually before signing.

Which regulations does the agent check?

The current rule packs cover GDPR (EU General Data Protection Regulation), DORA (EU Digital Operational Resilience Act), and DPDP (India's Digital Personal Data Protection Act 2023).

GDPR and DPDP are auto-detected from the region of each resource. DORA must be added manually in rules.yaml — it applies at the organisational level, not per-region.

We have a GPU instance — can we use our own LLM for triage?

Yes. Uncomment the triage_llm_url and triage_model fields in juro-deploy.yaml and point them at your Ollama endpoint:

triage_llm_url: "http://your-gpu-host:11434"
triage_model: "llama3.2:3b"

When these are set, juro-deploy init renders a three-service compose stack (scanner + RAG + your external LLM endpoint) instead of starting a local Ollama sidecar. The LLM is used only for advisory triage — it never affects the signed finding set.

What is the residency map, and do I need to review it before proceeding?

The residency map (residency_map.json) is a structured inventory of every resource the agent found during the discovery scan — grouped by service and region. No rule pack is active at this stage; it is purely a record of what exists.

You should review it before moving to Step 5 for two reasons: first, to confirm that all regulated workloads are present (if something is missing, widen the scope and re-scan); second, to give your compliance team an opportunity to confirm the resource list matches their expectations before the rule configuration is locked and signed.

How do I verify the signed artifacts later?

Once the standalone verifier ships, any party — including an auditor with no Juro account — will be able to verify any artifact using a single command:

juro verify /path/to/signing-bundle.json

The verifier checks that the config hash matches the file content and (once KMS signing is live) that the cryptographic signature is valid against the public key registered to your KMS key. It has no network dependency beyond optionally checking the transparency log entry.

The standalone verifier is in development. In the meantime, the config_hash field in the signing bundle can be independently verified with sha256sum.
